# 技術仕様書 (Architecture Design Document)

## テクノロジースタック

### 言語・ランタイム

| 技術 | バージョン |
|------|-----------|
| Rust | 1.83+ (Edition 2024) |
| Cargo | 1.83+ |

- Rust を選定した理由:
  - 単一バイナリとして配布でき、ユーザー環境にランタイム不要
  - C/C++並の実行速度で巨大ログファイルの処理に適する
  - メモリ安全性が保証され、セグフォやバッファオーバーフローのリスクがない
  - `BufReader` によるストリーミング処理でメモリ使用量を一定に保てる

## アーキテクチャパターン

### パイプライン・アーキテクチャ

logxはログファイルを入力として受け取り、パイプラインで処理する構造を採用する。
レイヤードアーキテクチャではなく、データフローに基づくパイプラインが自然。

```text
┌─────────────────────────────────────────────────────┐
│   CLIレイヤー（clap）                                │
│   引数パース・サブコマンドのディスパッチ              │
├─────────────────────────────────────────────────────┤
│   検出レイヤー（FormatDetector）                     │
│   ログフォーマットの自動検出                         │
├─────────────────────────────────────────────────────┤
│   パースレイヤー（LogParser trait）                  │
│   フォーマット別のログ行パース                       │
├─────────────────────────────────────────────────────┤
│   分析レイヤー（Analyzer）                          │
│   フィルタリング・集計・エラー判定                   │
├─────────────────────────────────────────────────────┤
│   表示レイヤー（Renderer）                          │
│   カラー出力・ヒストグラム・テーブル表示             │
└─────────────────────────────────────────────────────┘
```

#### CLIレイヤー

- **責務**: 引数のパース、ファイル存在確認、サブコマンドの実行
- **許可される操作**: 検出レイヤー・パースレイヤーの呼び出し
- **禁止される操作**: ログ行の直接パース（必ずLogParser経由）

#### 検出レイヤー

- **責務**: ファイル先頭のサンプリングによるフォーマット推定
- **許可される操作**: ファイルの先頭N行の読み取り
- **禁止される操作**: ファイル全体の読み込み

#### パースレイヤー

- **責務**: 1行をLogEntryに変換する。パース失敗時はNoneを返す
- **許可される操作**: 正規表現マッチング、JSON解析
- **禁止される操作**: ファイルI/O（行の文字列を受け取るだけ）

#### 分析レイヤー

- **責務**: LogEntryのフィルタリング・集計
- **許可される操作**: LogEntryの読み取り・集計
- **禁止される操作**: 出力処理（Renderer経由で行う）

#### 表示レイヤー

- **責務**: 結果の整形・カラー出力
- **許可される操作**: stdout/stderrへの書き込み
- **禁止される操作**: ビジネスロジック（判定・集計）

## データ永続化戦略

### ストレージ方式

logxはデータの永続化を行わない読み取り専用ツール。

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| ログファイル | ユーザーのファイルシステム | 各種ログフォーマット | 読み取り専用、変更しない |
| 設定ファイル（将来） | `~/.config/logx/config.toml` | TOML | XDG Base Directory準拠 |

### バックアップ戦略

- logxは読み取り専用ツールのためバックアップ不要
- ユーザーのログファイルは一切変更しない

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| コマンド起動（`logx --help`） | 50ms以内 | Apple M1, 8GB RAM |
| 100MBファイルのスキャン | 3秒以内 | Apple M1, 8GB RAM, SSD |
| 1GBファイルのスキャン | 30秒以内 | Apple M1, 8GB RAM, SSD |
| フォーマット検出（先頭20行） | 10ms以内 | Apple M1, 8GB RAM |

### リソース使用量

| リソース | 上限 | 理由 |
|---------|------|------|
| メモリ | 200MB | ストリーミング処理でファイルサイズに依存させない |
| CPU | 100%（1コア） | MVPではシングルスレッド処理 |
| ディスク | 0MB（一時ファイル不使用） | 読み取り専用のため |

## セキュリティアーキテクチャ

### データ保護

- **ファイルアクセス**: `O_RDONLY` で読み取り専用オープン。書き込みフラグは使用しない
- **メモリ保護**: 1行あたりのバッファ上限を1MBとし、異常な行によるOOMを防止
- **機密情報管理**: 将来のAI機能で使用するAPIキーは環境変数（`LOGX_API_KEY`）で管理

### 入力検証

- **ファイルパス**: パスの存在確認、通常ファイルであることの確認
- **CLIオプション**: clapによる型安全な引数パース（不正値は自動でエラー）
- **ステータスコード**: 100〜599の範囲チェック
- **Duration**: 正規表現で `\d+[smhd]` 形式を検証

### エラーハンドリング

- エラーメッセージにファイルの内容を含めない（ログに機密情報が含まれる可能性）
- パニックは使用しない。すべてのエラーは `Result` で伝播し、CLIレイヤーで処理

## スケーラビリティ設計

### データ増加への対応

- **想定データ量**: 最大10GBのログファイル
- **対策**: ストリーミング処理で1行ずつ処理。メモリにファイル全体を読み込まない
- **将来的な最適化**: メモリマップドI/O（mmap）の導入検討、マルチスレッド化

### 機能拡張性

- **LogParser trait**: 新しいフォーマット対応はtraitを実装するだけで追加可能
- **サブコマンド**: clapのサブコマンド機構で新機能を独立して追加可能
- **設定のカスタマイズ**: 将来的にTOML設定ファイルでカスタムフォーマット定義を対応

## テスト戦略

### ユニットテスト

- **フレームワーク**: Rust標準の `#[test]`
- **対象**: FormatDetector, 各LogParser実装, Analyzer, Renderer
- **カバレッジ目標**: 80%以上

### 統合テスト

- **方法**: `tests/` ディレクトリに配置、`tests/fixtures/` にサンプルログを用意
- **対象**: サブコマンド単位の入出力テスト

### E2Eテスト

- **ツール**: `assert_cmd` + `predicates` クレート
- **シナリオ**: CLIバイナリの実行、終了コード・stdout・stderrの検証

## 技術的制約

### 環境要件

- **OS**: macOS (Apple Silicon / Intel), Linux (x86_64 / aarch64), Windows (WSL2)
- **最小メモリ**: 64MB（logxプロセス自体）
- **必要ディスク容量**: 10MB（バイナリサイズ）
- **必要な外部依存**: なし（単一バイナリ）

### パフォーマンス制約

- MVPではシングルスレッド処理（将来的にRayonでパラレル化を検討）
- gzip/zstd圧縮ログの直接読み取りはMVPではスコープ外

### セキュリティ制約

- ネットワーク通信はMVPでは行わない（AI機能はPost-MVP）
- テレメトリ送信はMVPでは実装しない

## 依存関係管理

| ライブラリ | 用途 | バージョン管理方針 |
|-----------|------|-------------------|
| clap | CLIフレームワーク | メジャーバージョン固定（`4`） |
| regex | 正規表現 | メジャーバージョン固定（`1`） |
| chrono | 日時処理 | マイナーバージョン固定（`0.4`） |
| serde_json | JSON処理 | メジャーバージョン固定（`1`） |
| termcolor | カラー出力 | メジャーバージョン固定（`1`） |
| anyhow | エラー処理 | メジャーバージョン固定（`1`） |
| thiserror | エラー型定義 | メジャーバージョン固定（`1`） |

**方針**:

- `Cargo.lock` をリポジトリにコミットする（バイナリクレートのため）
- `cargo update` は手動で実行し、テストを通してからコミット
- セキュリティアドバイザリは `cargo audit` で定期チェック
