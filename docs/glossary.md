# プロジェクト用語集 (Glossary)

## 概要

このドキュメントは、logxプロジェクト内で使用される用語の定義を管理します。

**更新日**: 2026-03-08

## ドメイン用語

### ログフォーマット (Log Format)

**定義**: ログファイルの記述形式。各行の構造やフィールドの並び順を規定する。

**説明**: logxはフォーマットを自動検出し、適切なパーサーを選択する。対応フォーマットはApache Combined/Common、Nginx、JSON Lines、Syslog、PlainText。

**関連用語**: フォーマット自動検出, LogParser

### フォーマット自動検出 (Format Auto-Detection)

**定義**: ログファイルの先頭行を分析し、ログフォーマットを自動的に判定する機能。

**説明**: 先頭20行を読み取り、各フォーマットの正規表現パターンでマッチングを行う。適合率が最も高いフォーマットを採用し、50%未満の場合はPlainTextにフォールバックする。

**関連用語**: ログフォーマット, FormatDetector, 適合率

### 適合率 (Confidence)

**定義**: フォーマット自動検出において、サンプル行のうちパターンにマッチした行の割合。

**説明**: 0.0〜1.0の浮動小数点数。`マッチ行数 / サンプル行数` で算出する。

**使用例**:

- 適合率0.95 → 20行中19行がApache Combinedパターンにマッチ
- 適合率0.30 → PlainTextにフォールバック

### エラー抽出 (Error Extraction)

**定義**: ログファイルからエラーに該当する行を検出・抽出する機能。

**説明**: HTTPステータスコード(4xx/5xx)、ログレベル(ERROR/FATAL/CRITICAL)、PHPエラーパターン等を総合的に判定する。

**関連用語**: エラー判定, LogLevel

### スキャン (Scan)

**定義**: ログファイルの概要情報（行数、期間、ステータス分布、エラー率）を集計して表示する機能。

**関連用語**: ScanResult, ステータス分布

### フィルタリング (Filtering)

**定義**: 指定した条件（ステータスコード、ログレベル、時間範囲、パス、IP）でログ行を絞り込む機能。

**関連用語**: FilterOptions

## 技術用語

### clap

**定義**: RustのCLI引数パースライブラリ。deriveマクロで宣言的にCLIを定義できる。

**本プロジェクトでの用途**: CLIレイヤーのサブコマンド・オプション定義

**バージョン**: 4.x

### termcolor

**定義**: Rustのクロスプラットフォーム対応ターミナルカラー出力ライブラリ。

**本プロジェクトでの用途**: エラー行の赤色表示、ステータス分布の色分け等

**バージョン**: 1.x

### anyhow / thiserror

**定義**: Rustのエラーハンドリングライブラリ。thiserrorはライブラリ向け、anyhowはアプリケーション向け。

**本プロジェクトでの用途**: thiserrorでParseError等を定義、anyhowでCLIレイヤーのエラー処理

## 略語・頭字語

### MVP

**正式名称**: Minimum Viable Product

**意味**: 最小限の実用的なプロダクト。logxではF1〜F4（フォーマット検出、エラー抽出、スキャン、フィルタリング）がMVPスコープ。

### CLI

**正式名称**: Command Line Interface

**意味**: コマンドラインインターフェース。logxのユーザーインターフェース形態。

### CIDR

**正式名称**: Classless Inter-Domain Routing

**意味**: IPアドレスの範囲指定方式。`--ip "192.168.1.0/24"` のようにフィルタリングで使用。

## アーキテクチャ用語

### パイプライン・アーキテクチャ

**定義**: データが一方向に流れる処理パイプラインでシステムを構成するパターン。

**本プロジェクトでの適用**: CLI → FormatDetector → LogParser → Analyzer → Renderer の順にデータが流れる。

**関連コンポーネント**: FormatDetector, LogParser, Analyzer, Renderer

### ストリーミング処理

**定義**: ファイル全体をメモリに読み込まず、1行ずつ読み取りながら処理する方式。

**本プロジェクトでの適用**: BufReaderで1行ずつ読み取り、メモリ使用量をファイルサイズに依存させない（上限200MB）。

## データモデル用語

### LogEntry

**定義**: パース済みの1行分のログデータを表す構造体。

**主要フィールド**:

- `raw`: 元の行テキスト
- `timestamp`: 検出されたタイムスタンプ
- `status_code`: HTTPステータスコード
- `level`: ログレベル
- `path`: リクエストパス
- `ip`: クライアントIPアドレス
- `message`: ログメッセージ本文

**関連エンティティ**: LogFormat, LogLevel

### ScanResult

**定義**: `logx scan` の集計結果を格納する構造体。

**主要フィールド**:

- `total_lines`: 総行数
- `first_timestamp` / `last_timestamp`: 期間
- `status_distribution`: ステータスコード分布
- `error_rate`: エラー率

### LogLevel

**定義**: ログの重要度レベルを表す列挙型。

| レベル | 意味 | エラー判定 |
|--------|------|-----------|
| Debug | デバッグ情報 | No |
| Info | 通常情報 | No |
| Warn | 警告 | No |
| Error | エラー | Yes |
| Fatal | 致命的エラー | Yes |
| Critical | 重大エラー | Yes |

## エラー・例外

### ParseError

**発生条件**: ログ行のパースに失敗した場合

**対処方法**: その行をスキップし、処理を継続する。`--verbose` オプションで警告を表示。

### ファイルアクセスエラー

**発生条件**: 指定されたファイルが存在しない、または読み取り権限がない場合

**対処方法**: エラーメッセージを表示して終了。権限不足の場合は `sudo logx ...` を提案。

## 計算・アルゴリズム

### フォーマット検出スコアリング

**定義**: 各フォーマットのパターンマッチ率を計算し、最適なフォーマットを選択するアルゴリズム。

**計算式**:

```text
適合率 = マッチした行数 / サンプル行数（最大20行）
選択基準: 適合率が最大のフォーマット（50%未満はPlainText）
```

**実装箇所**: `src/detector.rs`

### エラー判定

**定義**: LogEntryがエラーかどうかを判定するアルゴリズム。

**判定基準**:

1. HTTPステータスコード >= 400
2. ログレベルが Error / Fatal / Critical
3. メッセージに "fatal error", "exception", "stack trace", "panic" を含む

**実装箇所**: `src/analyzer.rs`
