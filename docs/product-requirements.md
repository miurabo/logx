# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称

**logx** - 設定不要のログ調査CLIツール

### プロダクトコンセプト

- **ゼロ設定**: ログファイルを渡すだけで使える。フォーマットの指定や設定ファイルの作成は不要
- **誰でも使える**: grepやawkの知識がなくても、直感的なコマンドでログ調査ができる
- **高速処理**: Rust製で巨大ログ（数GB）もストレスなく処理できる

### プロダクトビジョン

ログ調査を「詳しい人に毎回聞く」作業から「誰でも自分でできる」作業に変える。CloudWatchやDatadogのようなSaaSを導入しなくても、ログファイルさえあれば即座に調査できるCLIツールを提供する。設定不要・学習コスト最小で、チーム全体のログ調査能力を底上げする。

### 目的

- ログ調査の属人化を解消する（特定の詳しい人への依頼集中をなくす）
- grep/tail/awkを知らないメンバーでもログ調査を自力で行えるようにする
- CloudWatch等のSaaS導入コストなしに、ログ調査の生産性を上げる

## ターゲットユーザー

### プライマリーペルソナ: 中村翔太（26歳、Webアプリケーションエンジニア）

- 経験3年目。PHP + Laravelで業務アプリを開発している
- 本番サーバーのログ調査が必要になるが、grepやawkの使い方がよく分からない
- エラーが起きると毎回先輩（ログに詳しい人）に調査を依頼している
- **現在の課題**: 「500エラーが出てるらしい」と言われてもログの見方が分からず、先輩の手が空くまで待つしかない
- **期待する解決策**: `logx errors /var/log/app.log` と打てばエラーだけ見られる。何が起きたかすぐ分かる
- **1日の典型的なワークフロー**: Slackで「エラー出てる」→先輩に頼む→待つ→教えてもらう → 本来は自分でやりたい

### セカンダリーペルソナ: 高橋洋介（35歳、インフラ兼バックエンドエンジニア）

- チームで一番ログに詳しい人。調査依頼が集中している
- Apache、Nginx、アプリログ、syslogなど複数種類のログを日常的に見ている
- 自分のgrep/awkスキルは高いが、毎回同じような調査を頼まれるのが負担
- **現在の課題**: ログ調査の依頼が1日3〜5件。自分の開発作業が中断される。メンバーに教えても定着しない
- **期待する解決策**: メンバーに「logx使って」と言えば自分で調査してもらえる。自分も複雑な調査はlogxで効率化したい
- **1日の典型的なワークフロー**: 開発中にSlackで調査依頼→sshしてgrep/tail→結果を共有→自分の作業に戻る（これを1日数回繰り返す）

## 機能要件

### コア機能（MVP）

#### F1: ログフォーマットの自動検出

**ユーザーストーリー**:
エンジニアとして、ログファイルのフォーマットを調べたり指定したりせずに、渡すだけで解析を始めたい

**受け入れ条件**:

- [ ] Apache Combined/Common形式を自動検出できる
- [ ] Nginx形式を自動検出できる
- [ ] JSON Lines（1行1JSON）形式を自動検出できる
- [ ] syslog形式を自動検出できる
- [ ] 未知のフォーマットの場合、プレーンテキストとして処理し、タイムスタンプのヒューリスティック検出を試みる
- [ ] 検出結果（使用フォーマット）をユーザーに表示する
- [ ] 検出精度: 上記4形式について95%以上の正解率（テストスイートで検証）

**優先度**: P0（必須）

#### F2: エラー抽出

**ユーザーストーリー**:
エンジニアとして、ログファイルからエラーだけを抽出して、何が起きているか素早く把握したい

**受け入れ条件**:

- [ ] `logx errors <file>` でエラー行を抽出して表示できる
- [ ] HTTPステータスコード（4xx, 5xx）をエラーとして検出できる
- [ ] ログレベル（ERROR, FATAL, CRITICAL）をエラーとして検出できる
- [ ] PHP Fatal error、Exception、stack traceを検出できる
- [ ] エラーの発生件数サマリーを表示できる（エラー種別ごとの件数）
- [ ] `--since <duration>` オプションで時間範囲を絞り込める（例: `--since 1h`, `--since 30m`）
- [ ] 出力はターミナルで見やすいカラー表示とする

**優先度**: P0（必須）

#### F3: ログスキャン（概要表示）

**ユーザーストーリー**:
エンジニアとして、ログファイルの全体像を素早く把握して、どんなログが記録されているか概要を知りたい

**受け入れ条件**:

- [ ] `logx scan <file>` でログファイルの概要を表示できる
- [ ] 表示項目: 総行数、期間（最初と最後のタイムスタンプ）、検出フォーマット
- [ ] ステータスコード分布（2xx/3xx/4xx/5xx の件数と割合）を表示できる
- [ ] エラー率（エラー行数 / 総行数）を表示できる
- [ ] 時間帯ごとのリクエスト分布（簡易ヒストグラム）を表示できる
- [ ] 複数ファイルを引数に渡せる（glob対応: `logx scan /var/log/app/*.log`）

**優先度**: P0（必須）

#### F4: フィルタリング

**ユーザーストーリー**:
エンジニアとして、特定の条件でログを絞り込んで、必要な情報だけを表示したい

**受け入れ条件**:

- [ ] `logx filter <file> --status 500` でHTTPステータスコードで絞り込める
- [ ] `logx filter <file> --level error` でログレベルで絞り込める
- [ ] `logx filter <file> --since 1h` で時間範囲で絞り込める
- [ ] `logx filter <file> --path "/api/users"` でリクエストパスで絞り込める
- [ ] `logx filter <file> --ip "192.168.1.0/24"` でIPアドレス/CIDR範囲で絞り込める
- [ ] 複数条件をANDで組み合わせられる
- [ ] 結果の件数をフッターに表示する

**優先度**: P0（必須）

### CLIインターフェース

```bash
# ログスキャン（概要表示）
logx scan /var/log/apache2/access.log
logx scan /var/log/app/*.log

# エラー抽出
logx errors /var/log/app/error.log
logx errors /var/log/app/*.log --since 1h

# フィルタリング
logx filter /var/log/apache2/access.log --status 500
logx filter /var/log/app.log --level error --since 30m
logx filter /var/log/apache2/access.log --path "/api/users" --status 5xx
```

### 将来的な機能（Post-MVP）

#### F5: AI による根本原因分析

`logx ask "500エラーが増えた原因は？" /var/log/app/*.log` のように自然言語で質問し、AIがログを分析して回答する。
ログのクエリストリングにパスワードが入っている場合など、秘匿情報をAIに送信しない仕組みがある。

**優先度**: P1（重要）

#### F6: リアルタイムモニタリング

`logx watch /var/log/app.log --errors` でログファイルをtailしながら、リアルタイムでエラーを検出・通知する。

**優先度**: P1（重要）

#### F7: レポート出力

分析結果をHTML/Markdownレポートとして出力し、チームに共有できる。

**優先度**: P1（重要）

#### F8: リモートサーバーのログ取得

SSH経由でリモートサーバーのログを取得・分析できる。

**優先度**: P2（できれば）

#### F9: Slack/Discord通知連携

エラー検出時にSlackやDiscordに通知を送る。

**優先度**: P2（できれば）

#### F10: カスタムフォーマット定義

ユーザー独自のログフォーマットを設定ファイルで定義できる。

**優先度**: P2（できれば）

## 非機能要件

### パフォーマンス

- コマンド起動時間: 50ms以内
- 100MBのログファイルのスキャン: 3秒以内
- 1GBのログファイルのスキャン: 30秒以内
- メモリ使用量: ファイルサイズに依存せず、ストリーミング処理で200MB以下

### ユーザビリティ

- 新規ユーザーが3分以内に `logx scan` と `logx errors` を実行できる
- すべてのサブコマンドに `--help` オプションを提供し、使用例を含む
- エラーメッセージには原因と推奨される対処法を含める
- ログファイルが空、または存在しない場合は分かりやすいエラーメッセージを表示する

### 信頼性

- ログファイルを読み取り専用で扱い、元ファイルを一切変更しない
- 不正なエンコーディングの行はスキップし、処理を継続する（エラーで停止しない）
- 壊れたJSON行はスキップし、処理を継続する

### セキュリティ

- APIキー等のトークンは環境変数で管理する（設定ファイルに平文保存しない）
- テレメトリはopt-in方式とし、デフォルトでは無効とする
- ログファイルの内容を外部に送信しない（AI機能を除く。AI機能もopt-in）

### 互換性

- macOS（Apple Silicon / Intel）、Linux（x86_64 / aarch64）で動作する
- Windows（WSL2環境）で動作する
- Rust 1.83以上でビルドできる
- 単一バイナリとして配布する（外部依存なし）

## 収益モデル

コア機能（F1〜F4）はOSSとして無料提供する。一部の拡張機能は有料プランで提供予定。

## スコープ外

明示的にスコープ外とする項目:

- **Web UI / ダッシュボード**: CLIツールに特化する。Web UIは提供しない
- **ログ収集・集約**: ログの収集やフォワーディングは行わない。既存のログファイルを分析対象とする
- **ログストレージ**: ログの保存・アーカイブ機能は提供しない
- **APM（Application Performance Monitoring）**: メトリクス収集やトレーシングは対象外
- **Windows ネイティブ対応**: WSL2経由でのみサポート。PowerShell/cmd.exeでの動作は対象外
